# xm.zlck888.com 运营中网站应急恢复方案

> 适用对象：`xm.zlck888.com` 项目申报系统  
> 备份基线：`/root/xm_clean_backup_2026-05-23_131421/`  
> 核心原则：**运营中优先恢复网站程序文件，不覆盖数据库，不覆盖新增上传资料。**

---

## 1. 适用场景

当出现以下情况时，可以按本方案处理：

- 首页、后台页面被篡改，出现非法跳转、博彩广告、恶意 JS。
- 文件中出现 `ccjs7f`、`tdcqt`、`xk.js`、`String.fromCharCode` 等异常特征。
- 后台入口异常，例如打开后台却跳到首页。
- ThinkPHP、`index.php`、`index.html`、`public/include/config.php` 等关键文件被污染。
- 验证码异常、后台登录页异常，但数据库业务数据仍正常。

---

## 2. 恢复原则

### 2.1 默认不要恢复数据库

网站正在运营中，项目申报数据、账号数据、填报记录通常都在数据库里。

因此：

- **只恢复网站文件**：通常不会影响备份之后新增的业务数据。
- **恢复数据库**：会覆盖备份时间之后新增/修改的数据。

除非确认数据库已经损坏或被篡改，否则不要执行数据库恢复。

### 2.2 保留上传目录

恢复程序文件时，应保留这些目录：

- `upload/`
- `uploads/`
- `excel/`

这些目录可能包含用户后续上传的附件、表格、材料。恢复时不要用旧备份覆盖它们。

### 2.3 先备份现场

处理前必须先备份当前异常现场，方便后续排查攻击入口，也避免误操作后无法回退。

## 4. 运营中推荐恢复命令

> 这是首选方案：只恢复程序文件，保留数据库和后续新增上传资料。

```bash
date_tag=$(date +%F_%H%M%S)
site=/www/wwwroot/xm.zlck888.com
bak=/root/xm_clean_backup_2026-05-23_131421
incident=/root/xm_incident_$date_tag
stage=/root/xm_restore_stage_$date_tag

mkdir -p "$incident" "$stage"

# 1. 解除关键文件锁，避免无法覆盖
chattr -R -i "$site" 2>/dev/null

# 2. 备份当前异常现场
tar --xattrs --acls -czf "$incident/webroot_before_restore.tar.gz" "$site"

# 3. 解压干净备份到临时目录
tar --xattrs --acls -xzf "$bak/xm.zlck888.com_webroot_final_digits_verify.tar.gz" -C "$stage"

# 4. 只恢复程序文件，保留运营中新上传的附件目录
rsync -a --delete \
  --exclude='upload/' \
  --exclude='uploads/' \
  --exclude='excel/' \
  "$stage/www/wwwroot/xm.zlck888.com/" "$site/"

# 5. 清理 ThinkPHP 缓存
find "$site/app/Runtime" -type f -delete

# 6. 检查并重载 Nginx
nginx -t && nginx -s reload
```

## 9. Nginx 防护规则

站点配置文件：

```bash
/www/server/panel/vhost/nginx/xm.zlck888.com.conf
```

应包含以下规则：

```nginx
location ~ ^/(app|ThinkPHP)/ {
    deny all;
}

location ~* ^/(upload|uploads|excel)/.*\.(php|phtml|php5|phar)$ {
    deny all;
}

location ~* ^/(lib|js|css|img|images|public)/.*\.(php|phtml|php5|phar)$ {
    deny all;
}
```

注意：

- 这些是 Nginx 配置语法，不能直接在 bash 里执行。
- `lib/`、`js/`、`css/`、`img/`、`images/`、`public/` 理论上应是静态资源目录，不应执行 PHP。
- 本次复发入口是 `/lib/img/_notes/databasse.php`，攻击者通过该 webshell 多次 POST 后污染了 `ThinkPHP/ThinkPHP.php`。
- 修改配置后必须执行：

```bash
nginx -t && nginx -s reload
```

配置文件建议锁定：

```bash
chattr +i /www/server/panel/vhost/nginx/xm.zlck888.com.conf
```

如需修改，先解锁：

```bash
chattr -i /www/server/panel/vhost/nginx/xm.zlck888.com.conf
```

- 当前白名单路径包含 `xm.zlck888.com`，应用到其他站点没有实际意义。

推荐：

- 先只保护 `xm.zlck888.com`。
- 其他重要站点逐个配置。
- 每个站点只白名单自己的缓存目录、上传目录、日志目录、`.well-known/`。

## 16. 常见问题

### 16.1 `tar: Removing leading '/' from member names` 是错误吗？

不是错误，这是 `tar` 的正常提示，表示压缩包内去掉了路径最前面的 `/`，避免误解压到绝对路径。

### 16.2 `sed: cannot rename ... Operation not permitted` 怎么办？

通常是文件或目录被 `chattr +i` 锁定了。先执行：

```bash
chattr -i 文件路径
```

修改后再锁回去：

```bash
chattr +i 文件路径
```

### 16.3 验证码不显示怎么办？

先检查验证码接口：

```bash
curl -s -D /tmp/verify.headers -o /tmp/verify.out \
'http://xm.zlck888.com/index.php?m=Admin&a=verify&t=123'

cat /tmp/verify.headers | grep -i "content-type"
file /tmp/verify.out
```

正常应返回 `image/png`。如果返回 HTML，查看错误日志：

```bash
tail -n 80 /www/wwwlogs/xm.zlck888.com.error.log
```

本次已修复的问题是 PHP 7.4 不兼容老 ThinkPHP 的 `String` 类名，最终验证码已改为纯数字。
