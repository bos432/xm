# 优化整改方案

## 0. 当前结论

本轮验收中新发现的权限可见问题已经修复并上线：

- 后台取消角色权限后，后端不再用内置默认权限补回。
- 部署 seed 不再覆盖已调整过的角色权限。
- 单位、区县、部门默认不再看到“专家认证”。
- 全周期页面副标题按当前账号可见页签动态展示。
- E2E 增加专家认证权限回归，最终结果 `9 passed`。

截至本次最终回归，没有发现需要继续阻塞交付的 P0/P1 整改项。以下为上线后优化建议。

## 1. 已完成整改

### 1.1 部署稳定性

- GitHub 仓库公开后，生产部署不再依赖 Token。
- 部署脚本完成前端构建、后端依赖安装、迁移、缓存刷新、Nginx reload。
- 部署后增加真实登录健康检查。
- `health_check_user` 已可通过 `health:login-check`。

### 1.2 首页与品牌素材

- 首页去除重复“单位注册”，保留“新单位注册”。
- 首页管理支持 Logo、Banner、Favicon。
- 未上传 Favicon 时返回 `favicon_url:null`，使用默认 `/favicon.ico` 兜底。

### 1.3 登录与安全

- 登录接口返回中文限流/锁定提示。
- 返回 `retry_after_seconds`，前端可显示倒计时。
- 安全中心记录登录限流事件。
- 支持白名单和临时放宽登录限流。

### 1.4 项目与全周期

- 项目列表支持 URL query 回显和筛选。
- 全周期支持通过 `/lifecycle?project_id=<id>` 自动回显项目。
- 合同任务书、实施进展、整改闭环、专家认证统一归入全周期。

### 1.5 验收管理

- 验收增加 `待处理 / 已处理 / 全部可见`。
- 区县、部门、专家可查看自己处理过的历史验收。
- 单位只查看本单位验收。
- 管理员和超管查看可见范围内验收。

### 1.6 测试数据治理

- 项目、验收、申报批次支持测试数据筛选。
- 识别规则：
  - `metadata.e2e=true`
  - 名称、编号、标题包含 `E2E-`
- 超管可归档测试批次。
- 不物理删除任何测试数据。
- 修复 `metadata` 为空时 `e2e=0` 误排除正式数据的问题。

### 1.7 E2E 自动化

- 已覆盖公开首页、六角色登录、项目筛选、全周期、验收历史、安全中心。
- 报告按运行批次输出到：

```text
modernization/e2e/reports/runs/<run-id>
```

## 2. 交付后建议立即做

1. 修改超级管理员默认密码。
2. 上传正式 Logo、Banner、Favicon。
3. 配置正式 SMTP 邮箱。
4. 录入正式申报批次。
5. 建立正式单位和审核人员账号。
6. 检查 Supervisor 队列常驻。
7. 做一次真实业务人工验收。
8. 由超管复核各角色权限矩阵，确认正式账号权限与业务分工一致。

## 3. P1 建议优化

### 3.1 Dashboard 统计口径开关

增加“是否纳入测试数据”开关：

- 默认排除测试数据。
- 超管可临时包含测试数据用于验收。

### 3.2 测试数据归档扩展

当前只归档测试批次。后续可增加：

- 测试项目归档视图。
- 测试验收归档视图。
- 全周期测试记录筛选。

仍建议不物理删除。

### 3.3 E2E 报告保留策略

建议：

- 保留最近 20 次报告。
- 超过保留范围自动压缩或清理。
- 部署日志输出本次 E2E 报告路径。

## 5. 风险提示

- 正式业务上线前不要删除 E2E 数据，否则自动化回归要重新建数据。
- 如果正式批次名称误含 `E2E-`，可能被识别为测试数据。
- 邮件 SMTP 如使用个人邮箱，需要开启授权码，不要使用登录密码。
- 登录风控不要关闭，只通过白名单和临时放宽辅助验收。
